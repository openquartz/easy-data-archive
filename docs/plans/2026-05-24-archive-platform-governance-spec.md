# 归档平台研发治理规范

## 1. 目标
- 为归档平台建立统一的测试门禁、集成验证标准和代码评审检查点。
- 确保所有功能上线前具备可验证、可审计、可回归的交付标准。

## 2. 测试策略

### 2.1 单元测试
- 范围：Service 层、规则编译、权限判断、速率计算、状态流转、告警判定、工具类。
- 要求：
  - 新增核心业务逻辑必须补充单元测试。
  - 正常路径、异常路径、边界条件至少各覆盖 1 类。
  - 涉及任务取消、权限控制、规则校验的逻辑必须覆盖失败场景。

### 2.2 集成测试
- 范围：认证登录、数据源测试、分组与规则持久化、任务触发、任务取消、日志查询、告警触发。
- 要求：
  - 必须覆盖数据库读写和事务边界。
  - 必须验证状态机流转正确。
  - 必须验证接口权限边界和错误码返回。

### 2.3 接口回归测试
- 范围：
  - 登录/登出
  - 数据源 CRUD 与测试连接
  - 分组树查询、规则 CRUD、规则校验
  - 手动触发任务、取消任务、任务详情、任务日志
  - 大盘汇总、告警查询
- 要求：
  - 关键接口全通过后方可提测。
  - 接口字段、状态码、业务码必须与文档一致。

### 2.4 前端验证
- 范围：登录、动态菜单、权限按钮、任务中心轮询、规则编辑、日志抽屉、监控图表。
- 要求：
  - 构建必须成功。
  - 关键页面在 Chrome 最新稳定版下交互正常。
  - 高危操作需有二次确认和权限控制。

## 3. 上线前通过标准
- 单元测试通过率：100%。
- 集成测试通过率：100%。
- 关键接口回归通过率：100%。
- 前端构建通过率：100%。
- 不允许带阻断级、严重级已知缺陷上线。
- 任务触发、任务取消、日志查询、权限校验必须完成预发验证。

## 4. 代码评审检查点

### 4.1 架构与规范
- 是否遵循既定模块边界和包结构。
- 是否复用已有领域模型和归档引擎抽象，而非重复造轮子。
- 是否保持命名、注释、响应结构、状态枚举统一。

### 4.2 安全
- 是否存在越权访问风险。
- 是否存在明文密码、敏感配置泄漏、日志打印敏感字段问题。
- 是否存在 SQL 注入、表达式注入、未校验输入风险。

### 4.3 业务正确性
- 任务状态流转是否完整。
- 规则优先级、分组启停、数据源状态、取消条件是否校验完整。
- 前后端状态枚举、字段语义、错误码是否一致。

### 4.4 数据库与性能
- 是否建立必要索引。
- 是否避免高频接口全表扫描。
- 日志、进度、告警类表是否考虑增长控制。
- 是否避免在事务中执行长耗时外部调用。

### 4.5 可观测性
- 是否补齐操作日志、任务日志、错误日志、告警记录。
- 是否提供明确的异常信息与排障上下文。
- 是否能追踪触发人、取消人、执行阶段、处理条数和速率。

### 4.6 测试与验证
- 是否补齐对应单元测试和集成测试。
- 是否提供必要的自测说明和验证证据。
- 是否验证失败场景和边界场景。

## 5. 发布建议
- 先灰度开放给平台管理员和归档管理员。
- 首批只开放手动触发，待稳定后再开放定时调度。
- 对任务取消、告警通知、日志查询重点观察一周。
