# UI Toast、可拉伸弹窗与归档分组明细预览设计

## 背景

当前 `easyarchive-ui` 中与后端交互失败后的提示主要以内联错误文案形式分散在各个页面和弹窗中，反馈位置不统一，也不适合高频操作场景。新增和编辑弹窗使用自定义 `modal-card` 容器，但尺寸固定，面对 SQL、备注等长内容时可读性不足。归档分组列表目前只有“编辑”和跳转详情页能力，缺少在列表上下文内快速查看分组明细的只读预览。

本次设计聚焦三个明确目标：

1. 所有后端交互失败信息统一通过 toast 弹窗展示，并在 5 秒后自动消失。
2. 管理端新增/编辑表单弹窗支持用户手动拉伸，以查看更多内容。
3. 在归档分组列表中，给每一行增加位于“编辑”旁的“详情”按钮，弹窗只读展示该分组下的两类明细列表。

## 范围

### 纳入范围

- `easyarchive-ui` 中通过 `http.ts` 发起的后端请求失败提示。
- 以下新增/编辑表单弹窗的可拉伸能力：
  - `ArchiveGroupFormDialog.vue`
  - `ArchiveGroupItemByTimeFormDialog.vue`
  - `ArchiveGroupItemByIdFormDialog.vue`
  - `DatasourceFormDialog.vue`
  - `UserFormDialog.vue`
  - `UserPermissionDialog.vue`
- 归档分组列表页 `ArchiveGroupView.vue` 中新增只读“详情”按钮与详情弹窗。

### 不纳入范围

- 本地表单校验错误的提示机制重构。此类错误继续保留在表单内联展示。
- 已存在的只读页面、详情页、通知面板、登录页等非新增/编辑表单弹窗。
- 归档分组详情弹窗中的编辑、删除、状态切换、跳转操作。
- 后端接口协议变更。

## 现状与问题

### 后端错误提示

当前 HTTP 请求在 `src/utils/http.ts` 中统一解析响应错误，但最终由各页面自行接住异常并写入 `errorMessage`、`actionErrorMessage` 等状态后以内联文本显示。结果是：

- 用户需要在页面局部区域寻找错误信息。
- 同一类错误在不同页面展示方式不一致。
- 新页面容易重复实现一套错误状态。

### 弹窗尺寸

当前表单弹窗统一使用 `modal-backdrop` + `modal-card` 结构，布局简洁，但尺寸固定。遇到以下内容时体验较差：

- `fetchSql`、`deleteWhere` 等多行 SQL 字段阅读空间不足。
- 备注和连接串字段较长时换行挤压布局。
- 用户无法主动拉宽或拉高弹窗。

### 分组明细预览

归档分组列表有“编辑”和跳转详情页的能力，但如果用户仅想确认该分组挂了哪些“按时间明细/按 ID 明细”，必须离开当前列表上下文，操作成本偏高。

## 方案对比

### 方案 A：基于现有自定义 UI 增量增强（推荐）

做法：

- 新增轻量级全局 toast 状态与渲染组件。
- 在 HTTP 层或统一错误入口触发 toast。
- 为目标表单弹窗增加统一的可拉伸样式封装。
- 新增归档分组明细只读预览弹窗组件，复用现有明细接口。

优点：

- 不引入新依赖，和现有代码风格一致。
- 改动集中，测试面可控。
- 能为后续页面复用统一交互模式。

缺点：

- 需要逐步把页面中的成功/失败提示职责梳理清楚。

### 方案 B：逐页面局部实现

做法：

- 各页面分别在 `catch` 中调用自己的 toast。
- 各弹窗单独写可拉伸样式。
- 归档分组页单独拼装详情弹窗。

优点：

- 初始落地快。

缺点：

- 错误提示逻辑继续分散。
- 后续维护成本高，样式一致性差。

## 推荐设计

采用方案 A，在不改变当前技术栈的前提下，补齐一层轻量级 UI 交互基础设施。

## 详细设计

### 1. 全局后端错误 toast

#### 目标

将所有后端交互失败信息提升为全局浮层反馈，默认展示 5 秒后自动消失，不再依赖页面内联错误区域才能看见。

#### 设计

- 新增通用 toast 状态管理模块，例如 `src/stores/toast.ts` 或 `src/utils/toast.ts`。
- 在 `App.vue` 或全局布局中挂载 `ToastContainer` 组件，负责渲染 toast 列表。
- toast 最少支持：
  - `type`: `error` / `success`
  - `message`
  - `durationMs`
  - `id`
- 默认位置使用右上角，避免遮挡主表格内容。
- 自动消失时间固定为 `5000ms`。
- 单次展示多条 toast 时按新增顺序纵向堆叠。

#### 错误触发策略

- 所有从 `http.ts` 冒泡出的后端错误，统一在最外层 toast 展示。
- 页面内如仍需保留错误状态，仅保留对布局必要的少量场景；本次纳入范围页面应移除对“后端错误必须显示为内联文案”的依赖。
- 表单本地校验错误不走 toast，继续在弹窗内联展示，避免用户提交前反复弹窗干扰。

#### 成功提示策略

- 创建、更新、删除、触发、取消等成功操作可同步迁移到同一 toast 体系。
- 成功 toast 同样自动 5 秒消失，减少页面顶部 `successMessage` 的存在必要性。

### 2. 新增/编辑弹窗支持拉伸

#### 目标

让用户可以手动扩大对话框宽高，便于查看长文本和多字段内容。

#### 设计

- 保持现有自定义弹窗结构，不替换成第三方组件。
- 为目标表单弹窗统一增加可拉伸容器样式，例如：
  - `resize: both`
  - 合理的 `min-width`、`min-height`
  - `max-width: min(96vw, ...)`
  - `max-height: 90vh`
  - `overflow: auto`
- 表单区域和页脚保持在弹窗容器内部滚动，不允许内容溢出到视口外无法操作。
- 对 SQL 文本域额外保留较高最小高度，保证拉伸后收益明显。

#### 适用弹窗

- 归档分组新增/编辑
- 按时间明细新增/编辑
- 按 ID 明细新增/编辑
- 数据源新增/编辑
- 用户新增/编辑
- 用户权限新增/编辑

### 3. 归档分组列表中的只读明细详情弹窗

#### 目标

在归档分组列表行内，提供快速查看该分组明细配置的只读入口，不离开列表上下文。

#### 入口与位置

- 在 `ArchiveGroupView.vue` 的操作列中新增“详情”按钮。
- 按钮位置放在“编辑”按钮旁边，保持操作分组一致。

#### 数据来源

- 使用现有归档明细接口：
  - `getArchiveGroupItemsApi(groupId)` 获取明细摘要列表。
- 弹窗内部将摘要数据按 `itemType` 分为两组：
  - `TIME`
  - `ID`

本次不额外调用单条详情接口，避免弹窗打开时串行请求过多；展示目标是“看列表”，不是“看单项完整表单值”。

#### 展示内容

- 弹窗标题：`归档分组明细 - {groupName}`
- 两个只读分区：
  - 按时间明细列表
  - 按 ID 明细列表
- 每个分区使用表格展示摘要字段，至少包括：
  - 源表
  - 目标表
  - 范围/区间
  - 优先级
  - 步长/批次量（如果摘要有值）
  - 写入开关
  - 清理开关
  - 状态
- 空状态需要明确提示“暂无按时间明细”或“暂无按 ID 明细”。

#### 交互约束

- 纯查看，不提供编辑、删除、启停、跳转入口。
- 弹窗支持关闭，不缓存跨会话脏状态。
- 如果加载失败，显示错误 toast，并在弹窗主体提供简洁失败占位或保持关闭。

## 数据流

### 后端错误 toast

1. 页面调用 `http.get/post/...`
2. `http.ts` 解析到接口失败或网络失败
3. 构造统一错误对象
4. 触发全局 `showToast({ type: "error", durationMs: 5000, ... })`
5. `ToastContainer` 渲染并自动移除

### 明细查看弹窗

1. 用户点击归档分组列表行“详情”
2. 页面记录当前分组、打开加载状态
3. 调用 `getArchiveGroupItemsApi(group.id)`
4. 按 `itemType` 分类为时间明细和 ID 明细
5. 弹窗只读渲染两个列表
6. 关闭时清理当前加载态和明细数据

## 错误处理

- 网络异常、401 以外的业务失败、接口返回 `code != success` 均通过错误 toast 展示。
- `401` 继续保留现有清理登录态逻辑，同时也允许向用户显示认证失效 toast。
- 明细弹窗打开失败时不允许出现静默失败。
- 本地表单校验错误不提升为 toast，避免误导为后端错误。

## 测试策略

### 单元/组件测试

- 覆盖 toast 自动消失与多条堆叠行为。
- 覆盖 HTTP 失败时能触发错误 toast。
- 覆盖归档分组明细弹窗对 `TIME` / `ID` 的正确分类展示。
- 覆盖两个分区空列表时的空状态展示。

### 现有测试更新

- 更新依赖旧内联 `successMessage` / `errorMessage` 文案的前端测试，改为断言 toast 或新的可见反馈。
- 若已有表单弹窗结构快照测试，需要同步接受可拉伸样式类名变化。

### 手工验证

- 在归档分组、数据源、用户、明细管理等页面触发后端错误，确认右上角出现错误 toast，5 秒自动消失。
- 打开所有纳入范围表单弹窗，确认可拖拽放大且内容仍可滚动。
- 在归档分组列表点击“详情”，确认弹出只读窗口，并分区显示按时间明细和按 ID 明细。

## 风险与边界

- 如果 toast 在 `http.ts` 直接触发，而页面层也手动触发，可能出现重复提示，需要统一单一出口。
- 若当前部分页面仍依赖内联成功/失败文案进行布局提示，迁移时要避免出现“toast 有了但旧状态没清干净”的双重反馈。
- 自定义弹窗启用 `resize` 后，不同浏览器对拖拽手柄样式支持略有差异，但不影响核心功能。

## 实施拆分建议

1. 先补齐通用 toast 基础设施并接入全局布局。
2. 再迁移纳入范围页面的后端成功/失败提示。
3. 接着给目标表单弹窗补统一可拉伸样式。
4. 最后实现归档分组列表“详情”按钮和只读明细弹窗，并补测试。
