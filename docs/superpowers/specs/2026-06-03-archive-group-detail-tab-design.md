# 归档分组详情内部标签页设计

## 背景

当前归档分组管理页面需要支持从分组列表进入独立的分组详情工作面。目标不是弹窗或抽屉，而是在平台主工作区内以内部标签页打开分组详情，承载当前分组的配置详情、归档分组明细以及已执行归档任务信息。

本设计面向现有 EasyArchive 平台结构，优先复用已有接口与页面模式，并将新增能力控制在“详情聚合页”范围内。

## 目标

1. 在归档分组管理列表点击“明细”后，打开平台内部新标签页展示分组详情。
2. 详情页以配置明细查看为主，任务执行信息为辅助。
3. 详情页同时展示分组基础信息、合并后的分组明细列表、执行概览与最近任务。
4. 支持同时打开多个分组详情标签页，并避免同一分组重复开页。

## 非目标

1. 不在本次设计中重做归档分组管理列表页整体布局。
2. 不在本次设计中展开任务明细全量内容，任务深层细节仍交给任务详情页。
3. 不在本次设计中拆分为“按 ID 归档”和“按时间归档”两个主工作区页面。

## 现状约束

### 已有后端能力

1. `GET /api/v1/archive/groups/{id}` 返回单个归档分组详情。
2. `GET /api/v1/archive/groups/{groupId}/items` 返回归档分组明细摘要，已将 `ID` 与 `TIME` 两类明细合并，并按优先级可排序展示。
3. `ArchiveGroupView` 已包含活动任务相关字段：
   - `activeTaskId`
   - `activeTaskStatus`
   - `activeTaskStartTime`
   - `canTrigger`
   - `canCancelActiveTask`
   - `canViewActiveTask`
4. 任务执行表已有 `ArchiveGroupExecuteTask` 与对应 Mapper，可支撑按分组聚合查询能力扩展。

### 缺失能力

1. 尚无专门面向“分组详情聚合页”的任务统计接口。
2. 现有单分组详情接口不适合继续膨胀成任务统计与任务列表的超大接口。
3. 详情页的内部标签页打开、复用与标题同步规则需要前端定义并实现。

## 用户决策记录

本设计基于以下已确认决策：

1. 明细通过平台主工作区内部标签页打开，而非浏览器原生新页签。
2. 详情页以“配置明细优先”为主，运行信息作为辅助区块。
3. 明细默认合并为一个列表，按优先级统一排序，通过类型标签区分 `ID` 与 `TIME`。
4. 已执行任务区域默认展示“关键统计 + 最近任务列表”。

## 方案比较

### 方案 A：单页纵向详情

将分组详情、明细列表、执行概览、最近任务全部按单页纵向堆叠。

优点：

1. 实现快。
2. 页面跳转成本低。

缺点：

1. 页面会较长，信息重心不够明确。
2. 任务区域容易冲淡“配置明细优先”的核心目标。

### 方案 B：双区块详情页

页面上半区聚焦分组详情与分组明细，下半区承载执行概览与最近任务。

优点：

1. 主次分明，符合“配置优先、任务辅助”的目标。
2. 能直接复用已有分组详情与明细列表接口。
3. 只需补充任务聚合接口，落地成本可控。

缺点：

1. 需要新增聚合接口与前端卡片级刷新逻辑。

### 方案 C：主详情 + 右侧任务侧栏

主体只展示配置与明细，任务放在右侧固定侧栏。

优点：

1. 配置阅读连续性强。

缺点：

1. 会压缩明细表格横向空间。
2. 不利于复杂字段展示与响应式适配。

### 结论

采用方案 B：双区块详情页。

## 信息架构

详情页内部标签标题：

`归档分组详情 - {groupName}`

页面从上到下分为五个区块：

1. 页头操作区
2. 分组详情卡片
3. 分组明细列表卡片
4. 执行概览卡片
5. 最近执行任务列表卡片

### 1. 页头操作区

展示：

1. 分组名称
2. 分组编码
3. 启用状态标签
4. 运行状态标签

操作：

1. `编辑分组`
2. `立即执行`
3. `查看运行中任务`

规则：

1. 若存在活动任务，在页头高亮运行状态标签。
2. 若 `canTrigger` 为 `false`，禁用 `立即执行`。
3. 若不存在活动任务，隐藏或禁用 `查看运行中任务`。

### 2. 分组详情卡片

默认两列信息栅格展示以下字段：

1. `分组ID`
2. `分组编码`
3. `分组名称`
4. `启用状态`
5. `备注`
6. `创建时间`
7. `更新时间`

若后续实体补充执行配置字段，例如并发数、调度策略或数据源标识，应放在本卡片内，不进入明细主表。

### 3. 分组明细列表卡片

这是详情页主区。

默认数据来源：

`GET /api/v1/archive/groups/{groupId}/items`

展示规则：

1. 合并展示 `ID` 与 `TIME` 两类明细。
2. 按 `priority` 升序排序。
3. 使用类型标签区分 `ID` 与 `TIME`，但不使用任务状态色。

默认列：

1. `优先级`
2. `明细类型`
3. `源表`
4. `目标表`
5. `步长`
6. `写入`
7. `清理`
8. `状态`
9. `操作`

字段映射：

1. `步长` 对应 `stepCount`
2. `写入` 对应 `enableWrite`
3. `清理` 对应 `enableClean`
4. `状态` 对应 `enableStatus`

操作建议：

1. `查看`
2. `编辑`
3. `启停`

说明：

主表保持摘要视图，不直接平铺 `fetchSql`、`deleteWhere`、时间字段名等复杂配置。点击 `查看` 后在当前详情页内以抽屉或局部面板展示完整明细配置，避免再打开新的主工作区标签页。

### 4. 执行概览卡片

展示当前分组任务统计摘要：

1. `累计执行次数`
2. `成功次数`
3. `失败次数`
4. `运行中次数`
5. `最近执行结果`
6. `最近执行时间`

该区域是辅助区块，视觉权重低于明细列表。

### 5. 最近执行任务列表卡片

默认展示当前分组最近 `10` 条任务。

默认列：

1. `任务ID`
2. `开始时间`
3. `结束时间`
4. `执行状态`
5. `处理记录数`
6. `处理速度`
7. `错误信息`
8. `操作`

展示规则：

1. `错误信息` 只展示摘要，超长省略，悬浮可查看全文。
2. `操作` 默认提供 `查看任务详情`。
3. 点击 `查看任务详情` 后再打开内部标签页，例如 `归档任务详情 - {taskId}`。

## 路由与内部标签页行为

### 路由建议

新增详情路由：

`/archive/groups/:id/detail`

### 打开规则

1. 在归档分组管理列表点击“明细”后打开该路由对应的内部标签页。
2. 若同一分组详情标签页已打开，则直接切换到现有标签页，不重复创建。
3. 支持多个不同分组详情页并存。

### 标签标题规则

1. 标签首次创建时使用临时标题 `归档分组详情`。
2. 分组详情加载成功后，更新为 `归档分组详情 - {groupName}`。
3. 如果分组名称被编辑并保存成功，同步刷新当前标签标题。

### 状态保留

每个详情标签页应独立保留以下状态：

1. 滚动位置
2. 明细筛选状态（若后续新增）
3. 最近任务分页或筛选状态（若后续新增）

## 数据加载与刷新策略

### 首屏加载

进入详情页后并行请求三块数据：

1. 分组详情
2. 分组明细列表
3. 执行概览与最近任务

### 刷新规则

1. `立即执行` 成功后，只刷新分组详情与执行概览/最近任务，不刷新明细列表。
2. `编辑分组` 成功后，刷新分组详情；若名称变化，同时更新标签标题。
3. 明细 `编辑` 或 `启停` 成功后，只刷新分组明细列表；若页面上方存在明细统计摘要，也一并刷新。
4. 任务列表点击 `查看任务详情` 时，不在当前页做深展开，转而打开任务详情标签页。

## 接口设计

### 复用接口

1. `GET /api/v1/archive/groups/{id}`
2. `GET /api/v1/archive/groups/{groupId}/items`

### 新增接口

新增聚合接口：

`GET /api/v1/archive/groups/{id}/overview`

职责：

1. 返回当前分组任务统计摘要。
2. 返回最近任务列表。
3. 可选返回明细数量摘要，避免前端重复计算。

建议响应结构：

```json
{
  "group": {
    "id": 1,
    "groupName": "订单归档",
    "groupCode": "ORDER_ARCHIVE",
    "enableStatus": 1,
    "activeTaskId": 101,
    "activeTaskStatus": 1,
    "activeTaskStartTime": "2026-06-03 10:00:00",
    "canTrigger": false,
    "canCancelActiveTask": true,
    "canViewActiveTask": true
  },
  "itemStats": {
    "totalCount": 12,
    "enabledCount": 10,
    "disabledCount": 2,
    "idTypeCount": 7,
    "timeTypeCount": 5
  },
  "taskStats": {
    "totalCount": 28,
    "successCount": 24,
    "failedCount": 3,
    "runningCount": 1,
    "lastExecuteStatus": 1,
    "lastExecuteTime": "2026-06-03 10:00:00"
  },
  "recentTasks": [
    {
      "id": 101,
      "groupId": 1,
      "startTime": "2026-06-03 10:00:00",
      "endTime": null,
      "executeStatus": 1,
      "processedRecords": 12000,
      "processedSpeed": 800,
      "errorMsg": null

```

### 为什么使用聚合接口

1. 保持 `GET /groups/{id}` 作为基础实体详情接口，不承担过多列表与统计职责。
2. 详情页可按卡片独立刷新，执行动作后只刷新 `overview` 即可。
3. 后续新增任务统计字段时，不会污染基础详情接口。

## 状态与异常处理

### 空状态

1. 分组不存在或已删除：整页显示“归档分组不存在或已被删除”。
2. 分组明细为空：明细区显示空表态，但分组详情和任务区正常显示。
3. 任务记录为空：执行概览统计归零，任务列表显示“暂无执行记录”。

### 运行态

1. 当存在活动任务时，在页头显示运行状态标签。
2. 高亮 `查看运行中任务`，便于快速跳转。

### 失败态

采用卡片级失败，不整页白屏：

1. 分组详情失败：页面主内容不可用，提示重试。
2. 明细列表失败：只在明细卡片内展示加载失败。
3. 任务概览失败：只在任务区域展示加载失败，不影响详情和配置阅读。

## 视觉与层级建议

1. 页头视觉权重最高，用于确认当前分组与可执行操作。
2. 分组详情卡片次高，用简洁信息栅格展示基础字段。
3. 明细列表作为主区，保证阅读密度与横向空间。
4. 执行概览与最近任务放在页面下半区，弱于明细区。

### 颜色规则

1. `启用`、`成功`：绿色
2. `停用`：灰色
3. `执行中`、`取消中`：蓝色或橙色
4. `失败`：红色

说明：

`ID/TIME` 类型标签使用中性色，不与任务状态语义色混用，避免误读。

## 实现建议

### 前端

1. 在归档分组列表的“明细”按钮接入内部标签页打开逻辑。
2. 新增分组详情视图页，对接三块数据加载。
3. 为内部标签页建立去重策略，键建议使用 `archive-group-detail:{groupId}`。
4. 将任务列表的 `查看任务详情` 接入现有任务详情工作区页面。

### 后端

1. 新增 `overview` 查询接口与 DTO。
2. 在任务 Mapper 中补充分组维度的任务统计与最近任务查询方法。
3. 复用现有 `ArchiveGroupView` 作为页头基础分组数据来源，避免重复装配。

## 验收标准

1. 在归档分组列表点击“明细”后，能在主工作区打开内部标签页。
2. 同一分组重复点击“明细”不会产生重复标签页。
3. 详情页可展示分组详情、合并后的明细列表、任务统计、最近任务。
4. 明细列表按优先级排序，类型为 `ID/TIME` 的明细能清晰区分。
5. 执行动作后，任务概览和最近任务可刷新反映新状态。
6. 任务区异常不影响分组详情和明细查看。

## 风险与后续事项

1. 若前端当前内部标签页框架尚未沉淀统一接口，需要先抽象标签页管理能力。
2. 若任务详情页尚未适配从分组详情跳转，需要补齐任务详情路由参数约定。
3. 若后续需要按明细维度查看执行历史，应另行设计“明细详情页”，不应继续向本页叠加复杂度。
