# 归档任务列表：分组名列与状态/分组筛选

日期：2026-06-27

## 背景

当前归档任务列表已具备基础的任务查询、状态字段与 `groupId` 字段，且已有
`ArchiveTaskGroupNameResolver.fillGroupNames(...)` 批量填充分组名的机制。
但前端列表尚未展示归档分组名列，筛选栏也缺少状态下拉与归档分组远程模糊
搜索。目标是在不引入任务表 join 分组表的前提下，补充分组名列与筛选能力，
并保持后端批量查询、不在 for 循环内查询数据库。

## 目标

- 归档任务列表增加“归档分组名”列。
- 筛选栏增加状态下拉与归档分组远程模糊搜索下拉。
- 任务查询只通过 `groupId` 与后端交互，不通过分组名文本查询。
- 后端继续遵循“不 join、不 for 循环内单条查询”的约束。

## 范围

### 范围内

- `easyarchive-starter` 中的任务查询服务层与控制器层。
- `easyarchive-ui` 中的任务列表视图、任务 API 层与 i18n 文案。
- 现有分组查询接口的复用方式与任务列表返回结构中 `groupName` 字段的填充
  策略。

### 范围外

- 任务表 SQL 不新增 join 分组表。
- 不修改归档分组的创建/编辑/删除逻辑。
- 不新增复杂的分组权限过滤逻辑，沿用现有管理员/非管理员查询路径。
- 不把分组名作为任务查询参数。

## 设计

### 架构与后端数据流

任务列表接口保持现有契约：

- 接口：`GET /api/v1/task-log/tasks`
- 参数：`page`、`size`、`status`、`groupId`

查询流程：

1. 服务层根据当前用户角色选择查询路径：
   - 管理员：`archiveLogRepository.queryTasks(...)`
   - 非管理员：`archiveGroupExecuteTaskMapper.selectPageByUser(...)`
2. 任务查询只使用任务自身字段过滤：
   - `execute_status`
   - `group_id`
   - `deleted = 0`
3. 查询得到分页任务后，服务层统一收集本页结果中的 `groupId` 集合。
4. 通过现有批量分组查询能力一次性获取分组信息。
5. 调用 `ArchiveTaskGroupNameResolver.fillGroupNames(...)` 将 `groupName`
   填充到任务对象。
6. 将任务对象转换为 `TaskVO` 返回，前端直接使用返回的 `groupName`。

关键约束：

- 不在任务 SQL 中 join `ea_archive_group`。
- 不在遍历任务列表时逐条查询分组。
- 非管理员路径同样在列表查询之后批量填充 `groupName`。

### 前端交互与组件

#### 筛选栏

在 `TaskListView.vue` 顶部筛选栏保留/新增两个筛选项：

- 状态下拉：传 `status`，空值表示全部状态。
- 归档分组远程模糊下拉：
  - 展示候选项的 `groupName`。
  - 用户输入关键字时调用分组分页接口 `GET /api/v1/archive/groups/page`
    并传入 `keyword` 进行远程模糊搜索。
  - 选中某项后，前端只保留 `groupId`，并把它作为 `groupId` 传给任务查询。
  - 清空选择时传 `undefined`，表示不按分组过滤。

#### 归档分组名列

在任务列表中增加“归档分组名”列：

- 优先展示后端返回的 `groupName`。
- 若 `groupName` 为空，则展示 `groupId`。
- 若 `groupId` 也为空，则展示 `-`。

#### API 层

复用现有 `getTasksApi(query: TaskQuery)`，参数继续只传 `page`、`size`、
`status`、`groupId`。

分组候选接口采用远程搜索：

- 调用分组分页接口并传入关键字参数。
- 候选接口失败时只影响分组下拉，不影响任务列表查询。
- 不在前端一次性加载全量分组后本地过滤。

### 错误处理与边界

- 分组候选搜索失败：
  - 不清空任务列表。
  - 前端保留当前已选分组或显示空候选，并给出轻量提示。
- 任务列表查询失败：
  - 沿用现有错误提示逻辑。
- 某个任务的分组已删除或查不到：
  - 后端返回空 `groupName`。
  - 前端展示兜底值（`groupId` 或 `-`）。
- 分组搜索无结果：
  - 下拉显示空结果。
  - 不把分组名文本作为查询参数传给任务接口。

### 测试与验证

#### 后端

调整/补充 `ArchiveTaskLogServiceImplTest`，重点验证：

- 按 `status` 过滤查询。
- 按 `groupId` 过滤查询。
- 返回结果中包含 `groupName`。
- 分组名通过批量方式填充，不存在 per-row 查询路径。
- 管理员与非管理员路径都能正确返回 `groupName`。

#### 前端

- 执行类型检查与构建。
- 手动验证：
  - 任务列表展示“归档分组名”列。
  - 状态下拉筛选生效。
  - 归档分组远程搜索生效。
  - 选中分组后任务列表只通过 `groupId` 过滤。
  - 清空筛选后恢复默认列表。
  - 分组名缺失/分组不存在时的兜底展示。

## 风险与权衡

- 若未来分组数量极大，远程搜索接口需要保持分页与关键字匹配性能；当前
  设计已避免一次性加载全量分组。
- 若任务列表分页后 `groupId` 集合过大，批量分组查询仍会增长参数规模；但
  单页任务数量有限，实际影响可控。
- 现有 `ArchiveTaskGroupNameResolver` 已具备批量填充机制，本次设计与其保持
  一致，避免引入新的数据访问模式。
