# EasyArchive UI 运维管理台设计

## 1. 背景与目标

当前仓库核心能力仍以 Java/Spring Boot 归档引擎为主，但 `easyarchive-starter` 已经暴露出可供管理台消费的一批平台 API，包括：

- 认证：`/api/v1/auth/login`、`/api/v1/auth/logout`、`/api/v1/auth/me`
- 数据源：`/api/v1/archive/datasources`
- 任务日志与取消：`/api/v1/task-log/tasks`、`/api/v1/task-log/tasks/{taskId}`、`/api/v1/task-log/tasks/{taskId}/logs`、`/api/v1/task-log/tasks/{taskId}/cancel`
- 用户：`/api/v1/users`

本次前端建设以“代码现状可支撑的能力”为边界，交付一个独立的 `easyarchive-ui` 模块，作为数据归档管理平台第一阶段的运维管理台。

## 2. 本阶段范围

### 2.1 在范围内

- 新增独立模块 `easyarchive-ui`
- 基于 `Vue 3 + TypeScript + Vite` 构建管理台
- 完成登录态闭环
- 完成工作台概览页
- 完成数据源管理页
- 完成归档任务列表页
- 完成任务详情与日志页
- 完成取消任务能力接入
- 完成用户管理页
- 为前端稳定消费补充必要的后端适配接口与响应一致性处理

### 2.2 明确不在范围内

- 归档分组管理
- 归档规则可视化建模
- 手动发起归档任务页面
- 完整 RBAC 权限配置页面
- 复杂 BI 图表、拖拽编排、向导式配置器

## 3. 交付目标

本阶段目标不是只做一个“能看”的前端壳，而是交付一个可登录、可查看任务、可管理数据源、可查看异常、可取消任务、可维护用户的稳定后台系统。

成功标准：

- 本地可独立启动 `easyarchive-ui`
- 前后端联调后形成完整登录闭环
- 核心列表页具备可用的分页、筛选、刷新与错误反馈
- 任务详情页能稳定展示任务状态、进度、错误与日志
- 写操作具备显式成功/失败反馈
- 常见联调问题有统一兜底机制，尽可能少出现页面白屏、无提示失败、状态错乱

## 4. 总体方案

采用独立前端模块方案：

- `easyarchive-ui` 作为仓库内新模块存在
- 开发期通过 Vite dev server 启动
- 与 `easyarchive-starter` 通过 HTTP API 联调
- 不嵌入 starter 静态资源目录，不与 Spring Boot 页面耦合

选择该方案的原因：

- 前后端职责清晰，便于独立开发与验证
- 不破坏现有 starter 启动形态
- 更适合后续继续扩展为完整平台
- 联调、构建和部署边界明确

## 5. 信息架构

管理台采用典型后台布局：

- 左侧固定主导航
- 顶部展示当前用户、登出入口、刷新入口
- 中间为高密度工作区

主导航固定为四项：

- 工作台
- 数据源
- 归档任务
- 用户管理

不加入未落地能力入口，避免出现“菜单先行、功能缺失”的伪完整平台。

## 6. 页面设计

### 6.1 登录页

职责：

- 用户名密码登录
- 保存 token
- 拉取当前用户信息
- 登录成功后进入工作台

交互要求：

- 表单校验完整
- 登录中禁用重复提交
- 登录失败时展示明确业务错误

### 6.2 工作台

职责：

- 展示任务状态统计
- 展示最近任务
- 展示异常任务
- 展示数据源状态概览

展现形式：

- 统计卡片
- 两个简洁表格区块
- 不引入复杂图表作为首版依赖

原因：

- 当前后端聚合能力有限
- 卡片 + 表格足以支撑运维值守场景
- 可显著降低图表适配和样式调优成本

### 6.3 数据源管理

职责：

- 数据源列表
- 新增数据源
- 编辑数据源
- 启停状态切换
- 连接测试

交互要求：

- 表单项显式校验
- 密码类字段避免在列表中明文展示
- 测试连接和保存动作分离
- 更新状态时有明确确认与反馈

### 6.4 归档任务

职责：

- 分页列表
- 按状态筛选
- 手动刷新
- 跳转任务详情

交互要求：

- 轮询刷新只在本页开启
- 支持空状态与加载态
- 状态标签统一映射

### 6.5 任务详情与日志

职责：

- 展示任务基础信息
- 展示处理条数、速度、心跳时间
- 展示错误信息
- 展示执行日志
- 在允许取消时提供取消入口

交互要求：

- 详情区与日志区分层明确
- 日志列表支持分页
- 取消任务前进行二次确认
- 取消请求提交后及时刷新状态

### 6.6 用户管理

职责：

- 用户列表
- 新增用户
- 编辑用户
- 状态切换

边界：

- 不做角色权限分配器
- 不做复杂组织树
- 保持基础运维能力即可

## 7. 前端技术方案

技术栈固定为：

- Vue 3
- TypeScript
- Vite
- Vue Router
- Pinia
- Axios
- Element Plus

目录建议：

- `src/api`：接口封装
- `src/types`：类型定义
- `src/stores`：登录态、全局字典、任务刷新状态
- `src/router`：路由与鉴权守卫
- `src/layouts`：主布局
- `src/views`：页面级视图
- `src/components`：通用组件
- `src/utils`：请求层、日期格式化、状态映射
- `src/styles`：主题变量与全局样式

## 8. 前端稳定性设计

为了尽可能少出现问题，前端必须具备以下稳定性约束：

### 8.1 统一请求层

- 统一封装 `Axios` 实例
- 自动注入 JWT
- 统一解包 `ApiResponse<T>`
- 统一处理 `401`、业务错误码、网络错误
- 统一弹出错误提示，避免页面各自处理

### 8.2 统一状态字典

将以下状态集中管理：

- 任务状态：等待、运行中、成功、失败、取消中、已取消
- 数据源状态：未测试、正常、异常
- 用户状态：启用、禁用

页面只消费统一字典，不直接写魔法数字。

### 8.3 受控轮询

- 仅任务列表页与任务详情页使用轮询
- 路由离开时自动销毁轮询
- 页面隐藏或卸载时停止刷新

### 8.4 表单防错

- 必填项与格式校验前置
- 提交中按钮禁用
- 避免重复点击导致重复写入

### 8.5 视图状态完整

所有主页面都必须覆盖：

- 首次加载态
- 空数据态
- 正常数据态
- 请求失败态

## 9. 视觉方向

视觉要求以简洁、高效、工程化为主：

- 浅色背景管理台
- 明确卡片层级
- 高信息密度但不拥挤
- 状态色统一
- 减少无意义装饰

设计不追求营销站式视觉表现，而强调：

- 运维场景下的信息可读性
- 列表与表单操作效率
- 长时间使用下的稳定观感

## 10. 后端适配需求

为支撑稳定交付，后端建议补充以下必要适配：

### 10.1 工作台概览接口

新增聚合接口，至少返回：

- 任务状态统计
- 最近任务列表
- 异常任务列表
- 数据源状态汇总

原因：

- 现有代码有任务与数据源原子接口
- 但工作台不应由前端自己拼过多请求与统计逻辑

### 10.2 认证闭环校验

需要确认并在必要时修正：

- `login` 响应结构是否稳定
- `/me` 是否可正确返回当前用户
- JWT 过滤器是否支持独立前端访问
- 退出登录行为是否符合前端消费预期

### 10.3 响应结构一致性

列表类接口统一按以下结构返回：

- `list`
- `total`
- `page`
- `size`

避免前端为不同接口写多套解析逻辑。

### 10.4 异常响应一致性

建议补一个全局异常处理，将未捕获异常统一转成：

- `code`
- `message`
- `data`

避免接口异常时返回 HTML 错页或不可预测结构。

### 10.5 CORS 与开发联调

如果当前 starter 未显式开放跨域，需要补充开发期跨域支持，以便 `easyarchive-ui` 在本地独立启动时可直接联调。

## 11. 非功能性要求

### 11.1 可维护性

- 页面、API、类型、状态分层清晰
- 尽量避免页面内混杂业务映射逻辑
- 公共逻辑优先抽离

### 11.2 可测试性

- 关键 API 封装可独立验证
- 关键状态映射与工具函数可单测
- 关键交互流程可做基础组件/页面测试

### 11.3 可扩展性

后续如补齐归档分组与规则管理，可在现有导航与目录结构下自然扩展，不需要推翻第一版架构。

## 12. 风险与控制

### 风险 1：后端接口虽存在，但返回语义不完全适合前端

控制方式：

- 实施前先做接口清单与类型对齐
- 必要时优先补后端适配层，不在前端堆特殊判断

### 风险 2：认证链路存在跨域或 token 处理问题

控制方式：

- 先打通登录与 `/me`
- 再推进其余页面

### 风险 3：任务轮询导致无效请求或状态抖动

控制方式：

- 将轮询集中封装
- 严格限制页面生命周期

### 风险 4：数据源与用户表单字段命名和数据库语义存在偏差

控制方式：

- 页面层不直接复用数据库术语
- 在 API/types 层做一次语义整形

## 13. 实施顺序建议

推荐顺序：

1. 搭建 `easyarchive-ui` 工程骨架
2. 打通请求层、登录态、路由守卫
3. 完成工作台概览后端适配与前端页面
4. 完成数据源管理
5. 完成任务列表
6. 完成任务详情与日志、取消任务
7. 完成用户管理
8. 做联调、错误态补齐与页面收口

## 14. 验证与验收

验收应覆盖以下场景：

- 登录成功 / 失败 / token 失效
- 工作台首次加载与异常反馈
- 数据源新增、编辑、状态变更、连接测试
- 任务列表分页、筛选、刷新
- 任务详情查看与日志查看
- 可取消任务的取消流程
- 用户新增、编辑、状态切换
- 断网、接口异常、后端业务错误时的统一提示

## 15. 结论

第一阶段采用“运维管理台”方案最稳妥。它严格以当前后端真实能力为边界，通过少量后端适配补齐前端消费面，能在风险可控的前提下交付一个完整且稳定的数据归档管理平台入口。
